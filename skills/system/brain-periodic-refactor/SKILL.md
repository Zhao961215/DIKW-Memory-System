# CIRAAF 大脑定期整理框架

> **定位**：大脑（Holographic fact_store）的领域级结构检查 + 重构定稿。
> 与已有机制的衔接详见 § 衔接设计。

---

## 流程概览

```
     ┌─────────────────────────────────────┐
     │      CIRAAF 大脑定期整理              │
     ├─────────────────────────────────────┤
     │                                     │
     │  P — 优先级排序：这轮先整哪个领域       │
     │       ↓                              │
     │  C — 分类：属于什么领域/性质           │
     │       ↓                              │
     │  I — 整合：全量扫描该领域facts          │
     │       ↓                              │
     │  R — 反思：找过期/矛盾/重复/断链/遗漏   │
     │       ↓                              │
     │  A — 完善：降权 + 更新 + 新增 + patch  │
     │       ↓                              │
     │  A — 整理：按 A/B/C/D/E 五层重构       │
     │       ↓                              │
     │  F — 定稿：MEMORY.md + 新fact + 告知   │
     │                                     │
     └─────────────────────────────────────┘
```

## 详细步骤

### P — 优先级排序（先确定这轮整哪个）

| 优先级 | 判断标准 | 例子 |
|--------|---------|------|
| 🔴 高 | 用户近期频繁问该领域问题，或体系刚发生过变化 | 投资逻辑（六风刚定型） |
| 🟡 中 | 距离上次整理超过4周，或者facts数量>该领域平均 | 系统知识 |
| 🟢 低 | 刚整理不久，facts数量少且稳定 | 用户偏好 |

**做法**：`fact_store(query="领域关键词", limit=50)` 查数量 → 对比上一次 `source="refactored:"` 的时间 → 决定优先级

### C — 分类（界定领域边界）

选定的领域有哪些子话题？预先划定搜索边界：

| 领域 | 包含子话题 | 搜索关键词覆盖 |
|------|-----------|---------------|
| 投资 | 体系框架/持仓/数据源/PE/动量/止盈/潜风/探风/挖呗 | PE,仓位,动量,止盈,探风,挖呗,潜风,乘风,换风,六风 |
| 系统 | Hermes架构/记忆系统/skill/配置/cron/工具 | hermes,记忆,holographic,skill,cron,config |
| 用户 | 偏好/风格/决策特点/沟通习惯 | 用户,偏好,风格,主上 |
| 开发 | 项目/DIKW/GitHub/设计 | 项目,github,设计,插件,dikw |
| 方法 | 元方法论/思维模型/分析框架 | 框架,方法论,原则,过程,元认知 |

### I — 整合（全量扫描该领域facts）

```python
# 核心动作：多维度搜
fact_store(query="PE 动量 止盈 仓位 基金 持仓", limit=50)
fact_store(query="探风 挖呗 潜风 乘风 换风 六风", limit=30)
fact_store(query="投资 原则 框架 体系 数据源", limit=30)
```

**输出**：该领域facts的完整清单，按created_at排序，标注信任分。

### R — 反思（找7类问题）

对整合结果逐条检查以下7类问题：

| 符号 | 问题类型 | 判断标准 | 处理 |
|------|---------|---------|------|
| 🗑️ | 过期 | 数据有明确时效（如持仓比例、净值快照），且已过有效期 | fact_feedback降权 |
| 🔄 | 逻辑矛盾 | 两条facts对同一事物做出相反断言 | 保留正确的，降权错误的 |
| 📋 | 重复 | 同一方法论/事实被记录多次 | 保留最新的，降权其余 |
| 📸 | 快照 | 临时动态快照（如"2026-05-15净值数据"），不应存在大脑里 | 降权，这类数据应走cache |
| 📦 | 被覆盖 | 旧框架/旧方法论已被新体系取代 | 降权，在新facts中写明"替代X" |
| 🕳️ | 遗漏 | 该有但没有的关键方法论 | 新写入 |
| 🔄 | **跨领域污染** | 当前领域包含明显属于其他领域的事实（如开发domain混入403条投资fact） | 改category到正确领域，tags标记cirAAF:分类修正 |

**输出**：问题清单（含fact_id + 问题描述 + 处理方案）

> **第二轮扫描建议**（2026-06-03实战验证）：首轮清除显性垃圾（碎片/重复/快照）后，在 trust=0.3-0.55 区间做第二轮扫荡。该区间是隐性垃圾的高发带——叙事快照、旧cron配置、auto-retain单字片段往往在此区间聚集。两轮合计可捕获比单轮多50-80%的问题。

### A1 — 完善：降权与更新

对反思发现的每条问题执行：

```json
// 过期/矛盾/重复/快照 → 降权
fact_feedback(action="unhelpful", fact_id=xxx)

// 遗漏/需要更新 → 写入
fact_store(content="新版方法论", query="...", tags="...")
```

**原则**：
- **不删除facts**（来时路也是我），通过降权让信任分排序自动过滤
- 发现矛盾时用 `source` 字段标记正确的版本（`source="refactored:2026-06-03"`）
- 每次标记完成后确认：`fact_feedback` 确实改变了信任分

### A2 — 整理：五层重构

将存活facts按五层重组：

| 层级 | 内容 | 标记 |
|------|------|------|
| **A类 — 体系框架** | 当前权威方法论（六风体系、总纲） | `source="refactored:日期"` |
| **B类 — 数据/标的** | 最新持仓、动态判断（食品饮料修正版） | 带时效标记，下次整理时核验 |
| **C类 — 工具/数据源** | PE数据源矩阵、接口连通性、工具关系 | 低频变动 |
| **D类 — 分析框架** | 行业生命周期、回测结论、理念原则 | 相对稳定 |
| **E类 — 已降权（历史参考）** | 0.6-0.8信任分，不主动用但可追溯 | 不修改，降权已自动处理 |

### F — 定稿（输出产物）

完成后输出三样：
1. **更新MEMORY.md**：定稿结论写入副文件索引（如果该领域方法论进入核心层）
2. **三条新fact（已被#7946-#7950覆盖）**：定稿已写
3. **告知用户**：变更摘要（清理了多少条、改了什么、新增了什么）

```text
P1.3 单条校准（SOUL铁律，日常）
  │
  ├── 发现单条fact过时 → 当场fact_feedback降权
  └── 发现体系层面的逻辑冲突 → 触发CIRAAF（完整跑一轮）
  
cron fact_feedback_loop.py（每周六11:00）
  │
  ├── 批量统计：降权0检索fact、升权高helpful率fact
  ├── 处理的是"单条fact的质量"→ 微观层
  └── CIRAAF处理的是"领域级结构一致性"→ 宏观层

CIRAAF（你当前在用）
  │
  ├── 处理的是"领域级的逻辑是否完整自洽"
  ├── 输出的是重构后的层次体系
  └── 三个工具的分工：微观日常 → 微观批量 → 宏观重构
```

**三者从微观到宏观形成三层保养流水线**：

```
P1.3 单条校准（日常对话中） 
    → fact_feedback 当场降权/升权
    ↓
Cron 统计降权（每周六 11:00） 
    → fact_feedback_loop.py 批量处理零检索/高helpful
    ↓
CIRAAF 体系重构（每2-4周/领域）
    → 纵向检查领域级的逻辑一致性 + 定稿
```

## 批量处理技巧（2026-06-03 实战经验）

对大型领域（≥1000条facts），用 `fact_store()` 逐条检索效率太低。推荐用 **SQLite 直连 + terminal** 批量扫描：

### 全量扫描脚本模式

```bash
cd ~/.hermes/hermes-agent && source venv/bin/activate && python3 << 'PYEOF'
import sqlite3
from pathlib import Path
db = Path.home() / '.hermes' / 'memory_store.db'
conn = sqlite3.connect(str(db))
conn.row_factory = sqlite3.Row
cur = conn.execute('''SELECT fact_id, content, tags, trust_score 
    FROM facts WHERE category IN ('general','reflect') 
    ORDER BY trust_score ASC''')
all_facts = [dict(r) for r in cur.fetchall()]
# ... 后续分析
PYEOF
```

### 批量降权/升权

SQLite 直接 UPDATE 比逐个 fact_feedback 快上百倍。⚠️ 仅在 LLM 驱动的 CIRAAF（Gear 2）中使用，Gear 1 保持纯源码操作：

```python
conn.execute("UPDATE facts SET trust_score = 0.05, 
    updated_at = ? WHERE fact_id = ?", (now, fid))
conn.commit()
```

### 各领域洁净度经验（2026-06-03 四领域实测）

| 领域 | 事实数 | 洁净度 | 特点 |
|------|--------|--------|------|
| 用户(user_pref) | 442 | ⭐⭐⭐ | 100%人工录入，0碎片/0重复 |
| 系统(system/tool) | 1196 | ⭐⭐ | 少量重复，无碎片污染 |
| 开发(project/decision/discovery) | 3071 | ⭐⭐ | 过载但无碎片，易混入投资类fact（403条） |
| **方法(general/reflect)** | 639 | ⭐ | **最脏**——auto-retain的垃圾桶 |

**根因**：Hindsight（已停用）的 auto-retain 将未匹配到具体 category 的事实默认写入 `general`/`reflect` 类别。Holographic 本身无 auto-retain/auto-reflect 功能，所有事实均需显式 `fact_store` 或代码写入。**方法领域的碎片全部来自 Hindsight 时期，Holographic 运行期间不产生新碎片。**

### 快速识别碎片公式（直接降权类）

| 条件 | 案例 | 处理 |
|------|------|------|
| 长度≤5字符且全标点/数字 | `"4.5"`、`"|"`、`"{}"` | 🗑️ 降权 |
| 纯数字/短日期 | `"2025-01-13"`、`"0.1"` | 🗑️ 降权 |
| 以 `` { [ ` // `` 开头且<80字符 | `{"title":...}`、`//api.xxx` | 🗑️ 降权 |
| 固定 cron 噪音 | `"此对话作为定时 cron 任务运行"` | 🗑️ 降权 |
| 以 `"reflect: 发现关联"` 开头且 t<0.6 | 自动关联模板噪音 | 🗑️ 降权 |

## ⚙️ 自动化实现（2026-06-03 源码落地）

> CIRAAF 已从纯手工流程升级为**三齿轮架构**：Gear 1 和 Gear 3 完全零模型依赖，Gear 2 仍需 LLM 但通过 cron 自动调度。

### 架构总览

```
┌────────────────────────────────────────────────────────┐
│           CIRAAF 自动化流水线（无LLM依赖比例 ~66%）      │
└────────────────────────────────────────────────────────┘
                         │
     ┌───────────────────┼───────────────────┐
     ▼                   ▼                   ▼
┌──────────┐     ┌──────────────┐     ┌──────────┐
│ Gear 1    │     │ Gear 2       │     │ Gear 3    │
│ 机械层     │ ──→ │ LLM 反射层    │ ──→ │ 应用层     │

├──────────┤     ├──────────────┤     ├──────────┤
│ 健康监控   │     │ 矛盾检测      │     │ 执行修复   │
│ 统计衰减   │     │ 五层分类      │     │ 标记定稿   │
```

### Gear 1 — `agent/cirAAF_mechanic.py`（源码级，零 LLM）

| 功能 | 命令 | 频率 |
|------|------|------|
| 健康报告 | `python3 -m agent.cirAAF_mechanic` | 按需 / cron 每周日 10:00 |
| 机械衰减（降权陈旧事实） | `python3 -m agent.cirAAF_mechanic --decay` | DB>60天后每周 |
| 反射数据包生成 | `python3 -m agent.cirAAF_mechanic --refactor-report` | 每月（给 Gear 2 用） |
| 单领域详细扫描 | `python3 -m agent.cirAAF_mechanic --domain 投资` | 按需 |

**5 大领域定义**（硬编码在 `DOMAINS` 字典）：

| 领域 | 关键词 | category 匹配 | 健康分(当前) | 上次整理 |
|------|--------|--------------|-------------|---------|
| 投资 | PE,仓位,动量,止盈,探风... | investment, project | 63 | 2026-06-03 |
| 系统 | hermes,memory,holographic... | system, tool | 62 | 从未 |
| 用户 | 用户,偏好,风格... | user_pref | 64 | 从未 |
| 开发 | 项目,github,设计,插件... | project, decision, discovery | 60 | 从未 |
| 方法 | 框架,方法论,原则... | general, reflect | 57 | 从未 |

**衰减条件**（三条件全部满足才执行，各领域独立判断）：
1. 该领域有 **≥5 条事实 trust > 0.7** → 表明手动升权形成了信任分层
2. 该领域健康分 **连续2周未改善** → 表明停滞，需要机械清理推一把
3. 目标事实年龄 > **该领域中位年龄 × 1.5** → 动态阈值，适配不同领域成熟度

**触发流程**：每周日 cron 跑 `--decay` 时自动检查三条件 → 不满足则跳过并打印原因 → 满足才执行。完全不需要人记时间。

### Gear 2 — LLM cron（仍需模型推理）

当 Gear 1 的 `--refactor-report` 发现某领域健康分 < 60 时，触发一个 cron job：
- 加载 `brain-periodic-refactor` skill
- 读取 Gear 1 输出的 JSON 数据包
- 执行 R（反思）：找矛盾/过期/重复/遗漏
- 执行 A1（完善）：输出修复指令 JSON
- 指令格式：
  ```json
  [
    {"action": "unhelpful", "fact_id": 123},
    {"action": "add", "content": "...", "tags": "refactored:投资", "category": "investment"},
    {"action": "update_tags", "fact_id": 456, "tags": "refactored:投资,体系"}
  ]
  ```

### Gear 3 — 指令应用（源码级，零 LLM）

```bash
python3 -m agent.cirAAF_mechanic --apply-refactor /path/to/fix_instructions.json
```

执行 Gear 2 输出的所有修复指令（降权/新增/标记/更新信任分）。

### Cron 定时任务

| 任务 | 时间 | 类型 | 说明 |
|------|------|------|------|
| CIRAAF 周健康报告 | 每周日 10:00 | no_agent=True | 脚本直出健康报告到飞书 |
| Gear 2 月反射 | (待DB>60天后配置) | LLM cron | 自动触发低健康分领域重构 |
| Gear 3 指令执行 | Gear 2 完成后 | no_agent=True | 自动执行修复 |

### 与两层校准的关系

```text
P1.3 单条校准（微观，日常对话中 → 模型手动）
    │ 发现错误事实 → 当场 fact_feedback
    ▼

    ▼
      三层从微观到宏观形成完整保养闭环
```

```bash
# 3. 第一个整完的领域是"投资"，下次建议："方法"（健康分最低57）
```
