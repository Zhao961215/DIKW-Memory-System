> 本文件定义**记忆系统层**（知识/经验），面向使用 Holographic 记忆引擎的 Hermes Agent。
> 模型层（天赋/能量）在 config.yaml，Agent 层（人格/规则）在 SOUL.md。
>
> **适用场景**：Holographic 是 Hermes Agent 内置的 8 种记忆插件之一，
> 零外部服务依赖，纯 SQLite + FTS5 + HRR 代数本地运行。无 embedding 外部依赖，
> 不会出现 hindsight 那种 new-api 500 错误。

## 一、一句话说清楚

**大脑存高频，图书馆存低频。** 这里的大脑是 Hermes 官方 Holographic 插件——使用 HRR（Holographic Reduced Representations）代数 + FTS5 全文索引进行双引擎检索，全本地运行，零网络依赖。

| 哪儿 | 叫什么 | 用来存什么 | 访问方式 |
|------|--------|-----------|---------|
| 🧠 **大脑** | Holographic 🛠️ `fact_store` / `fact_feedback` | 方法论、原则、红线，需要跨 session 持久化的内容 | `fact_store` 工具检索 / 写入 |
| 📚 **图书馆** | 知识库 Vault（`vault/`） | 偶尔翻阅的参考文档 | `read_file`，稍慢 |
| 🗂️ **卡片柜** | 实体页 Entities（`vault/entities/`） | 实体/概念的卡片信息 | `read_file` 直接读 |
| 📦 **仓库** | Data Cache（`data/{模块}/cache/`） | 原始数据、临时缓存 | 不直接访问，需要时抓取 |
| 🏃 **速查** | `session_search` | 近期对话历史 | 搜跨 session 上下文 |

## 二、Holographic 是什么——检索机制详解

### 能力一览
| 项目 | 说明 |
|------|------|
| 存储 | ✅ `fact_store` — 保存事实到 SQLite（自动创建 FTS5 全文索引 + HRR 向量） |
| 检索 | ✅ **双重机制**：每次轮次前自动 FTS5 检索 + `fact_store` HRR probe 显式查询 |
| 反馈 | ✅ `fact_feedback` — 给事实打分（`action='helpful'` / `action='unhelpful'`） |
| 去重 | ✅ 自动去重，根据内容 UNIQUE 约束合并 |
| 持久化 | ✅ **跨 session 全局共享**到 `~/.hermes/memory_store.db` |

### 跨 session 全局共享（重要——与 Hindsight 不同）

Holographic 的 `memory_store.db` 是所有 session 共享的单库，**不按 session_id 做数据隔离**：

- `add_fact()` 写入时不关联 session_id
- `search()` / `probe()` 检索时不过滤 session_id
- session A 存的原则，session B 的 `prefetch`（自动 FTS5）和显式 `fact_store(query=...)`（HRR probe）都可以命中

**这意味着**：
1. ✅ **不需要固定 session ID**（DeepSeek 那套建议对应的是 Hindsight/Mem0 这类有 session 隔离的 provider）
2. ✅ 跨 session 记忆开箱即用
3. ⚠️ 存了不相关/低质量的事实会影响所有 session 的检索质量——确保 `query` 编码准确、`fact_feedback` 及时校准

### 检索机制：FTS5 + HRR 双引擎

Holographic 的检索不是只有 HRR，而是三层配合：

```
  自动层（每轮注入）：FTS5 全文索引
  ─────────────────────────────────
  Hermes 内核在每次轮次前自动调用 prefetch()，
  用 FTS5 关键词匹配事实库，将相关事实注入上下文。
  所以即使你不显式调用 fact_store，模型也能感知到相关记忆。

  显式层Ａ（关键词检索）：FTS5 search_facts
  ─────────────────────────────────
  fact_store(action='search') 时，同时走 FTS5 关键词命中 +
  Jaccard 相似度重排 + trust 加权。

  显式层Ｂ（代数检索）：HRR probe
  ─────────────────────────────────
  fact_store(action='probe') 时，通过循环卷积计算实体与
  事实库之间的代数相关性，找到结构上相关的记忆。
```

| 场景 | 哪个引擎在工作 | 效果 |
|------|--------------|------|
| 用户提到之前聊过的投资原则 | FTS5 自动检索 | ✅ 自动注入上下文 |
| 你想主动查"建仓策略" | FTS5 + Jaccard + HRR 混合 | ✅ 显式搜索 |
| 用户说"上次那件事"但记不清关键词 | 先 FTS5 + 再 HRR probe | ⚠️ 关键词广谱度决定命中率 |
| 用户说"保守点入场"但存的是"高估不建仓" | HRR 可能不匹配 | ❌ 关键词设计决定 |

### 检索流水线细节

```
fact_store(query="关键词")
  ↓
① FTS5 候选（limit × 3）：全文索引命中
  ↓
② Jaccard 重排：查询与事实内容的词元重叠度
  ↓
③ HRR 代数相似度：编码文本 → 循环卷积比对
  ↓
④ 加权融合：score = FTS5×0.4 + Jaccard×0.3 + HRR×0.3
  ↓
⑤ 信任加权：final = score × trust_score
  ↓
⑥ 返回 top-N
```

权重默认值：FTS5=0.4, Jaccard=0.3, HRR=0.3。可配置（见「配置调优」章节）。

### 不需要语义搜索的原因

HRR + FTS5 + Jaccard 的组合已经覆盖了 **80% 以上的检索场景**：

- FTS5 自动注入让你不需要每次都显式搜索
- Jaccard 重排弥补 FTS5 词序敏感的短板
- HRR 代数比纯关键词更灵活（对词序不敏感、支持部分匹配）
- 唯一覆盖不到的是**字面完全不重叠但语义相近**的查询——这通过正确编码关键词（见「HRR 关键词编码策略」）就能大幅缓解

**这对应的是设计取舍，不是缺陷**：Holographic 在 ARM64 轻量设备上，加入语义搜索意味着引入深度学习模型（bge-m3 / Ollama），额外 1-2GB 内存 + CPU 负载。在 x86 本机上也是同样道理——HRR 用纯代数运算替代语义模型，0 额外开销，不会因为 embedding 模型挂掉导致整个记忆系统瘫痪。

### 唯一真正的局限：无内置 Reflect

Holographic 没有像 Hindsight 那样的自动反思/整合机制。
需要外部 cron 来定期整理事实库（矛盾检测、信任分下调等）。
但如果只是作为"存方法论+检索"的轻量记忆系统，Holographic 够用。

## 三、核心触发时刻——分析完成后自动分流

> 这是整套系统的**真正触发点**，不是等用户说"存一下"才动手。

**每次用户引导你完成一项分析/工作后——无论是讨论一个业务逻辑、debug 一个系统问题、还是梳理一个知识框架——自动执行以下操作：**

1. **归纳总结**：这次分析中使用了什么方法、逻辑框架、判断依据？把"怎么做的"提炼出来
2. **DIKW 分流**：将过程中产生的信息按四层模型分别处理（见下表）
3. **不需要问用户"要不要存"**——方法论直接进大脑，这是你的职责，不是他的负担

### 具体实操：分析完成后自问三句

```
① 这次分析用了什么方法/框架？→ 存大脑（fact_store）
② 过程中产生了什么结构化数据？→ 存 entities/（write_file）
③ 有没有完整的分析报告？→ 存 vault/（write_file）
```

## 四、DIKW 四层模型——判断信息该放哪

```
┌─────────────────────────────────────┐
│         WISDOM（智慧）                │
│  不变的原则、框架、方法论              │
│  → 图书馆（完整版）+ 大脑（索引摘要）   │
├─────────────────────────────────────┤
│         KNOWLEDGE（知识）             │
│  实体信息、行业定义、概念解释            │
│  → 实体页（entities/）                │
├─────────────────────────────────────┤
│       INFORMATION（信息）              │
│  业务指标、快照数据——会变的动态数据      │
│  → Data Cache（带时间戳，30天失效）    │
├─────────────────────────────────────┤
│          DATA（数据）                  │
│  原始 API 返回、未处理的缓存            │
│  → Data/Cache 目录（临时）             │
└─────────────────────────────────────┘
```

**核心判断工具——问三个问题：**

| 问题 | 答案决定存哪 |
|------|-------------|
| 这个信息会变吗？ | **会变** → data cache（带时间戳） / **不会变** → 大脑或图书馆 |
| 我每次推理都要用吗？ | **高频** → 大脑（Holographic） / **偶尔查** → 图书馆（Vault） |
| 它是一张卡片还是一本书？ | **卡片**（单个实体）→ 实体页 / **体系化知识** → 图书馆 |

> ⚠️ 常见错误：把动态数据（快照、指标）塞进大脑。正确做法：动态数据进 data cache，大脑只存**不变的方法论**。
> ⚠️ Holographic HRR 检索天然对短文本更友好。长文本存前先提炼成**一句话原则**再存。

## 五、信息来了怎么办——三步分流

### Step 1：判断信息类型

```
信息进来
  │
  ├── 原始数据（未处理的API返回、日志）─────→ Data 临时目录
  │
  ├── 动态数据（指标/快照/短期判断）───────→ Data Cache（带时间戳）
  │
  ├── 结构化卡片（实体/概念/配置）────────→ 实体页 entities/
  │
  ├── 方法论/原则/红线（高频不变）────────→ 大脑 Holographic（高频用）
  │
  └── 完整报告/深度分析/部署文档 ─────────→ 图书馆 Vault（低频查）
```

### Step 2：检查有没有重复

- **有同类信息** → 更新/覆盖，不重复创建
- **没有** → 新建

### Step 3：执行存储

| 目标 | 用什么工具 | 示例 |
|------|-----------|------|
| 大脑 Holographic（存） | `fact_store(content="...", query="...", source="...")` | 存方法论、原则 |
| 大脑 Holographic（搜） | `fact_store(query="关键词")` — FTS5 + HRR 混合检索 | 搜历史记忆 |
| 大脑 Holographic（反馈） | `fact_feedback(action='helpful'/'unhelpful', fact_id=...)` | 校准信任评分 |
| 实体页 | `write_file` → `entities/{实体名}.md` | 写实体卡片 |
| 图书馆 | `write_file` → `vault/分类/文档名.md` | 写完整分析报告 |
| Data Cache | `write_file` → `data/{模块}/cache/` | 存临时数据 |

## 六、运行时检索——层级递进流水线

> 与"会话初始化"不同：初始化是冷启动补全背景，检索是每次用户提问时逐层查找。

用户提问时，按以下 **6 层递进检索，命中即返回，不继续下探**：

```
用户提问
  ↓
🏃 第 0 层：指代词快速路径
  "刚才/之前/上一条/你刚才说/刚刚" → session_search
  ↓
🧠 第 1 层：Holographic 双重检索
  自动：Hermes 内核每轮前自动 FTS5 检索注入（全局共享库，跨 session 命中）
  显式：fact_store(query="关键词1 关键词2") — HRR probe
  → 命中即返回
  ↓ 未命中
📦 第 2 层：缓存点检查
  检查 data/{模块}/cache/ 文件 mtime + TTL
  → 详见 cache-check skill
  ↓ 失效/无缓存
📚 第 3 层：知识库（Vault + 实体页）
  read_file entities/ → read_file vault/
  ↓ 未命中
🔍 第 4 层：近期对话
  session_search(关键词) — 换同义词再试
  ↓ 未命中
🌐 第 5 层：网络搜索
  web_search —— 最后手段
```

> **HRR 检索不命中时不要放弃**：换同义词再试 1-2 次。
> 不同关键词组合 = 不同的 HRR probe 结果，范围广的关键词更容易命中。

### 第 0 层：指代词快速路径 🏃
```python
if any(w in user_query for w in ["刚才", "之前", "上一条", "你刚才说", "刚刚"]):
    session_search(user_query)
    return
```

### 第 1 层：Holographic 检索 🧠
```
fact_store(query="关键词1 关键词2")
```
返回结果包含：`fact_id`, `content`, `trust_score`, `created_at`（跨 session，无隔离）。
→ 命中 → 回答
→ 未命中 → 换同义词再试，仍不中 → 进入第 2 层

> ⚠️ **Holographic 的检索局限**：HRR 代数是符号匹配，不是语义理解。
> 存了"市场高估时不建仓"，搜"牛市该不该买"可能不命中。
> **建议存事实时使用广谱关键词**（见「HRR 关键词编码策略」章节），让 HRR 更容易匹配到。
>
> ⚠️ **FTS5 中文分词硬伤**：FTS5 默认 unicode61 tokenizer 不识别中文词语边界，纯中文长句（连续汉字无空格/英文/数字/标点）被当作一个 token。这让中文 `fact_store(query=...)` 检索可能返回空。
> **对策**：搜中文方法论时如果 `fact_store` 返回空，不硬试同义词，直接跳到第 4 层 `session_search` 兜底。
> **预防**：存新事实时，content 用短句 + 空格分隔关键词（如"高估 不建仓 等待 保守"），确保 FTS5 可索引。

### 第 2～5 层
（同通用版，略）

### 六条检索原则

1. **指代词优先**：含指代词先查近期对话，不绕圈子
2. **记忆优先**：之前聊过的事先查 Holographic，不瞎猜
3. **缓存优先**：刚抓过的数据先看 data cache，不重复抓
4. **实体页优先**：已知实体名先看 entities/，命中即秒回
5. **时效检查**：读 data cache 时要核实时效性（见 `cache-check` skill）
6. **诚实原则**：找不到就说找不到，不编造。Holographic 检索可能漏召回，可以再试 `session_search`

## 七、Holographic 插件使用详解

### 配置（config.yaml）
```yaml
plugins:
  hermes-memory-store:                          # ← 注意：key 名是 hermes-memory-store
    db_path: ~/.hermes/memory_store.db
    auto_extract: 'false'                       # 依赖 SOUL.md 三问流程，不自提取
    default_trust: '0.5'                        # 新事实信任分（默认 0.5，中性推荐）
    min_trust_threshold: '0.3'                  # 检索过滤下限（默认 0.3）
    temporal_decay_half_life: '0'               # 关闭时间衰减（方法论不过期）
    hrr_weight: 0.15                            # HRR 检索权重（中文场景推荐 0.15）
    # hrr_dim: 1024  ← 默认值，无需显式设置
```

> ⚠️ **配置 key 注意**：Holographic 的配置键是 `plugins.hermes-memory-store`，
> 不是 `plugins.holographic`。这是 Hermes 内置插件系统的统一命名。
激活后，`fact_store` 和 `fact_feedback` 两个工具自动注册到工具列表。

### `fact_store` 工具
**用途**：存储事实 + 检索记忆

**存储模式**：
```json
{
  "action": "fact_store",
  "parameters": {
    "content": "市场全面高估时不建仓，等机会",
    "query": "建仓 高估 时机",
    "source": "投资原则"
  }
}
```
- `content`：要存的事实内容（必填）
- `query`：HRR probe 关键词，也用于后续检索匹配（建议填多个同义词）
- `source`：来源标签（可选，如"投资原则"、"系统运维"）

**检索模式**：
```json
{
  "action": "fact_store",
  "parameters": {
    "query": "高估 不建仓 保守"
  }
}
```
返回匹配的事实列表。

### `fact_feedback` 工具
**用途**：对检索结果进行反馈，校准信任分

```json
{
  "action": "fact_feedback",
  "parameters": {
    "fact_id": "xxx-xxx-xxx",
    "action": "helpful"
  }
}
```
- `action`: `helpful` / `unhelpful`

### `MemoryStore` 内部接口（实战必读）

> 直接绕过 `fact_store` 工具、用 `MemoryStore` Python API 处理批量/迁移/恢复任务。

```python
from plugins.memory.holographic.store import MemoryStore
ms = MemoryStore()
```

**关键方法**：
- `ms.add_fact(content, category='general', tags='')`：写入（自动算 hrr_vector 如可用）
- `ms.update_fact(fact_id, ...)` / `ms.remove_fact(fact_id)`：CRUD
- `ms.list_facts(category='', limit=10)` / `ms.search(query, limit=10)`：检索
- **`ms.rebuild_all_vectors(dim=None) -> int`**：**用于迁移/恢复**——遍历所有 facts 调 `_compute_hrr_vector` + `_rebuild_bank`，返回处理条数
- `ms._hrr_available: bool` / `hrr._HAS_NUMPY: bool`：**模块级缓存**——改 numpy 后必须 `kill 7` 才生效

**`rebuild_all_vectors()` 典型使用场景**：
1. **首次启用 HRR**：venv 装好 numpy + `kill 7` 后，旧 fact 没 hrr_vector → 调一次回填
2. **numpy 装/卸切换**：跨重启用 HRR 的 fact 重新生成向量
3. **DB 迁移/恢复**：从备份还原后确保所有 hrr_vector 与当前 hrr_dim 一致
4. **`hrr_dim` 变更**：从 1024 改 512（或反之）需要全部重算

```python
# 标准回填
ms = MemoryStore()
n = ms.rebuild_all_vectors()
print(f"处理 {n} 条")

# 变更维度（罕见）
ms.rebuild_all_vectors(dim=512)  # 会修改 self.hrr_dim
```

**何时不该调**：单条 fact 写入走 `add_fact` 已经自动算 hrr_vector，**不要**为单条写入调 rebuild（浪费）。

## 八、Holographic 配置调优

> Holographic 默认权重（FTS5=0.4, Jaccard=0.3, HRR=0.3）适合通用场景。
> 如果搜索效果不理想，可以在配置层面调整。

### 字段含义（实战必读）

| 字段 | 默认值 | 作用 | 对应运行时表现 |
|------|--------|------|---------------|
| `hrr_dim` | `1024` | HRR 向量维度 | `hrr_vector` 字段 BLOB 大小 = `hrr_dim × 8`（float64 字节数）→ 1024 = 8192B |
| `hrr_weight` | `0.3` | HRR 检索权重（中文场景建议 `0.15`）| score = FTS5×0.4 + Jaccard×0.3 + HRR×hrr_weight |
| `default_trust` | `0.5` | 新事实初始信任分 | 影响 trust 加权后的 final score |
| `min_trust_threshold` | `0.3` | 检索过滤下限 | trust < 0.3 的事实不参与召回 |
| `auto_extract` | `'false'` | 是否自动提取 fact | 关掉，靠 SOUL DIKW 三问手动分流 |
| `temporal_decay_half_life` | `0` | 时间衰减半衰期（天）| 0 = 不过期（方法论场景） |

**`hrr_dim` 实战对应**：
- 默认 1024 → `hrr_vector` 8192B = 1024 float64
- 改 512 → 4096B（节省空间，召回略差）
- 改 2048 → 16384B（更精确，4x 内存）
- 改完调 `ms.rebuild_all_vectors(dim=新值)` 重算

**验证字段对应**：直接 sqlite 查
```sql
SELECT length(hrr_vector), count(*) FROM facts WHERE hrr_vector IS NOT NULL GROUP BY length(hrr_vector);
```
应只返回一行 `(8192, N)`（假设默认 1024 维）。

### 权重含义

| 权重 | 默认值 | 作用 | 配置键 |
|------|--------|------|--------|
| FTS5 | 0.4 | 关键词全文检索（精确命中） | ❌ 不可配置（硬编码） |
| Jaccard | 0.3 | 词元重叠度重排（容错模糊匹配） | ❌ 不可配置（硬编码） |
| HRR | 0.3 | 代数相关性检索（灵活近义匹配） | ✅ `plugins.holographic.hrr_weight` |

> ⚠️ 与 DeepSeek 建议的区别：Holographic 没有 `textWeight` / `vectorWeight` / `hybrid_weight` / `retrieval_strategy` / `minScore` 这些配置项——那是 Hindsight 或其他 provider 的。Holographic 的实际可调项只有 `hrr_weight`。

### 调优指引

**场景 1：中文文档/精确查找**
→ 降低 `hrr_weight`（减小代数匹配的噪声）
```yaml
plugins:
  hermes-memory-store:
    hrr_weight: 0.15   # FTS5 + Jaccard 占主导
```

**场景 2：灵活近义搜索（口语化/模糊描述）**
→ 保持或提高 `hrr_weight`
```yaml
plugins:
  hermes-memory-store:
    hrr_weight: 0.4    # HRR 代数匹配占比更高
```

**场景 3：NAS 无 numpy 时自动降级**
如果 numpy 不可用，HRR 权重**自动归零**，并重新分配给 FTS5=0.6, Jaccard=0.4：
```
无 numpy → hrr_weight=0.0, fts_weight=0.6, jaccard_weight=0.4
```
此时配置中的 `hrr_weight` 设置会被忽略，检索完全依赖 FTS5 + Jaccard。如果发现搜索效果变差，先确认 numpy 是否安装。

### DeepSeek 建议中不适用于 Holographic 的项

| 建议 | 为什么不适用 |
|------|-------------|
| 固定 session ID | Holographic 全局共享，无需 session 隔离 |
| `retrieval_strategy: hybrid` | 无此配置项 |
| `minScore` 阈值调低 | 无此配置项 |
| `hermes memory rebuild --force` | Holographic 无向量索引可重建 |
| 迁移到向量数据库 (ChromaDB/FAISS) | NAS 资源有限，HRR 策略够用 |
| 集成 GHRR | 纯学术概念，无 Holographic 实现 |

> **总结**：Holographic 的调优核心只有**两个有效杠杆**：
> 1. `hrr_weight` — 权重分配
> 2. **HRR 关键词编码质量**（见下一章）——这才是真正起决定性作用的

## 九、HRR 关键词编码策略

> ⚠️ 以下是最关键的优化点——**HRR 检索的成败 90% 取决于 query 参数的编码质量**。
> 编得好，HRR 效果接近语义搜索；编得差，存进去就找不回来了。

### 核心公式

```
query = 核心概念 + 同义词群 + 近义词群 + 相关场景词
```

### 好例子 vs 差例子

| | query 参数 | 效果 |
|--|-----------|------|
| ❌ | `"建仓"` | 用户只说"进场"就搜不到 |
| ✅ | `"建仓 买入 入场 加仓 进场 开仓 买进"` | 无论哪种说法都能命中 |
| ❌ | `"高估"` | 搜"估值过高 泡沫 贵了"都不中 |
| ✅ | `"高估 估值过高 泡沫 太贵 不便宜 高估值"` | 全面覆盖 |

### 分类编码模板

**投资原则类：**
```python
query = " ".join([
    # 核心概念
    "建仓 加仓 买入 入场",
    # 反面词（HRR 能识别反面关系）
    "不减仓 不卖出 持有",
    # 条件/场景
    "时机 等待 确认 信号 右侧 左侧",
    # 情绪/判断
    "保守 谨慎 观望 耐心",
])
```

**用户偏好类：**
```python
query = " ".join([
    "飞书 报告 推送 通知",
    "偏好 喜欢 习惯 方式",
    "接收 发送 格式 排版",
])
```

**系统运维类：**
```python
query = " ".join([
    "Docker 容器 部署 启动 重启",
    "端口 网络 连接 日志 错误",
    "排查 修复 调试 监控",
])
```

### 关键词编码要诀

1. **广度优先**：宁多勿少，10 个同义词比 3 个精确词好
2. **场景覆盖**：把可能问到的场景词都放进去
   - 存"高估不建仓" → query 也加"卖 减仓"（因为用户可能聊到卖出时关联到这个原则）
3. **使用自然语言中的常用词**：用户大概率怎么说，就怎么编
4. **存事实时顺便编 query**：每次 `fact_store` 时，花 3 秒想一想"以后我会怎么搜这个"
5. **定期补 query**：发现某个事实总搜不到，用 `fact_feedback` + 重新 `fact_store` 更新 query

### 检索时的技巧

```
搜不到的时候，换以下思路各试一次：
1. 缩小范围：加更多限定词
   fact_store(query="建仓 买入 等待")
2. 扩大范围：去限定词，留核心词
   fact_store(query="建仓")
3. 完全换一套词：换个角度描述同一概念
   fact_store(query="入场 时机 右侧")
```

## 十、时效性管理——什么信息会过期

| 信息类型 | 有效期 | 过期后怎么办 |
|---------|--------|------------|
| 实时数据 | 数小时 | 重新抓取/计算 |
| 周频数据 | 7 天 | 重新获取 |
| 快照数据 | 30 天 | 重新确认 |
| 方法论/原则 | 永久 | 不需要更新 |
| 实体基本信息 | 更新时覆盖 | 手动或 cron 更新 |

**实操规则**：从 Holographic 或 data cache 读到动态数据时，先问"这个数据是多久前的？" 超期就重新获取。

## 十一、三条铁律

### 铁律 1：不重复
同一信息只存一处。lib 引用代替复制。

### 铁律 2：检索回退链
```
fact_store(query) — FTS5 + Jaccard + HRR 混合（最常用）
  → 查缓存点 → 知识库
  → session_search（近期对话）
  → web_search → 老实说找不到
```

### 铁律 3：宁多勿少
拿不准要不要存，先存。`fact_feedback` 可以校准信任分。

## 十二、Holographic vs Hindsight 对比

| 维度 | Holographic | Hindsight |
|------|-------------|-----------|
| 架构 | Hermes 内置插件 | 独立 daemon + PostgreSQL |
| 依赖 | 零外部服务 | Docker + PG |
| ARM64 | ✅ 原生支持 | ❌ 自建镜像卡死 |
| 部署方式 | `plugins.memory_provider=holographic` | Docker compose |
| 检索方式 | FTS5 + Jaccard + HRR 代数 | 语义向量 + BM25 + 图谱 |
| 语义搜索 | ❌ 不支持（设计取舍） | ✅ 支持 |
| 工具 | `fact_store` / `fact_feedback` | `hindsight_recall` / `retain` / `reflect` |
| 跨 session | ✅ 全局共享，开箱即用 | ⚠️ 按 bank_id 隔离 |
| 去重 | ✅ 自动 | ✅ 自动 |
| Reflect | ❌ 无内置 | ✅ `hindsight_reflect` |
| DB 位置 | `~/.hermes/memory_store.db` | PostgreSQL（远程） |

## 十三、一句话总结

> **Holographic 轻量够用：零依赖跑在 NAS 上，FTS5 自动检索打底 + Jaccard 重排 + HRR probe 显式召回。全局共享跨 session，不需要固定 session ID。调优的两个杠杆：hrr_weight 权重 + query 关键词编码质量。同义词围得够广，HRR 就够用。**
