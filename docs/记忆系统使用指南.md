# 记忆系统使用指南（Holographic 版）

> 本文件定义**记忆系统层**（知识/经验），面向使用 Holographic 记忆引擎的 Hermes Agent。
> 模型层（天赋/能量）在 config.yaml，Agent 层（人格/规则）在 SOUL.md。
>
> Holographic 是 Hermes Agent 内置的 8 种记忆插件之一，零外部服务依赖，纯 SQLite + FTS5 + HRR 代数本地运行。

---

## 一、一句话说清楚

**大脑存高频，图书馆存低频。** 这里的大脑是 Hermes 官方 Holographic 插件——使用 HRR（Holographic Reduced Representations）代数 + FTS5 全文索引进行双引擎检索，全本地运行，零网络依赖。

| 哪儿 | 叫什么 | 用来存什么 | 访问方式 |
|------|--------|-----------|---------|
| 🧠 **大脑** | Holographic 🛠️ `fact_store` / `fact_feedback` / `fact_feedback_loop.py` | 方法论、原则、红线，需要跨 session 持久化的内容 | `fact_store` 工具检索/写入；自动校准由 `fact_feedback_loop.py` 处理 |
| 📚 **图书馆** | 知识库 Vault（`vault/`） | 偶尔翻阅的参考文档 | `read_file`，稍慢 |
| 🗂️ **卡片柜** | 实体页 Entities（`vault/entities/`） | 实体/概念的卡片信息 | `read_file` 直接读 |
| 📦 **仓库** | Data Cache（`data/{模块}/cache/`） | 原始数据、临时缓存 | 不直接访问，需要时抓取 |
| 🏃 **速查** | `session_search` | 近期对话历史 | 搜跨 session 上下文 |

## 二、Holographic 是什么——检索机制详解

### 能力一览
| 项目 | 说明 |
|------|------|
| 存储 | ✅ `fact_store` — 保存事实到 SQLite（自动创建 FTS5 全文索引 + HRR 向量） |
| 检索 | ✅ **双重机制**：每次轮次前自动 FTS5 检索 + `fact_store` HRR probe 显式查询 |
| 反馈 | ✅ `fact_feedback` — 给事实打分（`action='helpful'` / `action='unhelpful'`） |
| 去重 | ✅ 自动去重，根据内容 UNIQUE 约束合并 |
| 持久化 | ✅ **跨 session 全局共享**到 `~/.hermes/memory_store.db` |

### 跨 session 全局共享（重要——与 Hindsight 不同）

Holographic 的 `memory_store.db` 是所有 session 共享的单库，**不按 session_id 做数据隔离**：

- `add_fact()` 写入时不关联 session_id
- `search()` / `probe()` 检索时不过滤 session_id
- session A 存的原则，session B 的 `prefetch`（自动 FTS5）和显式 `fact_store(query=...)`（HRR probe）都可以命中

**这意味着**：
1. ✅ **不需要固定 session ID**
2. ✅ 跨 session 记忆开箱即用
3. ⚠️ 存了不相关/低质量的事实会影响所有 session 的检索质量——确保 `query` 编码准确、`fact_feedback` 及时校准

### 检索机制：FTS5 + HRR 双引擎

Holographic 的检索不是只有 HRR，而是三层配合：

```
  自动层（每轮注入）：FTS5 全文索引
  ─────────────────────────────────
  Hermes 内核在每次轮次前自动调用 prefetch()，
  用 FTS5 关键词匹配事实库，将相关事实注入上下文。
  所以即使你不显式调用 fact_store，模型也能感知到相关记忆。

  显式层Ａ（关键词检索）：FTS5 search_facts
  ─────────────────────────────────
  fact_store(action='search') 时，同时走 FTS5 关键词命中 +
  Jaccard 相似度重排 + trust 加权。

  显式层Ｂ（代数检索）：HRR probe
  ─────────────────────────────────
  fact_store(action='probe') 时，通过循环卷积计算实体与
  事实库之间的代数相关性，找到结构上相关的记忆。
```

| 场景 | 哪个引擎在工作 | 效果 |
|------|--------------|------|
| 用户提到之前聊过的投资原则 | FTS5 自动检索 | ✅ 自动注入上下文 |
| 你想主动查"建仓策略" | FTS5 + Jaccard + HRR 混合 | ✅ 显式搜索 |
| 用户说"上次那件事"但记不清关键词 | 先 FTS5 + 再 HRR probe | ⚠️ 关键词广谱度决定命中率 |
| 用户说"保守点入场"但存的是"高估不建仓" | HRR 可能不匹配 | ❌ 关键词设计决定 |

### 检索流水线细节

```
fact_store(query="关键词")
  ↓
① FTS5 候选（limit × 3）：全文索引命中
  ↓
② Jaccard 重排：查询与事实内容的词元重叠度
  ↓
③ HRR 代数相似度：编码文本 → 循环卷积比对
  ↓
④ 加权融合：score = FTS5×0.4 + Jaccard×0.3 + HRR×0.3
  ↓
⑤ 信任加权：final = score × trust_score
  ↓
⑥ 返回 top-N
```

权重默认值：FTS5=0.4, Jaccard=0.3, HRR=0.3（hrr_weight 可配置）。

## 三、核心触发时刻——分析完成后自动分流

> 这是整套系统的**真正触发点**，不是等用户说"存一下"才动手。

**每次用户引导你完成一项分析/工作后——无论是讨论一个业务逻辑、debug 一个系统问题、还是梳理一个知识框架——自动执行以下操作：**

1. **归纳总结**：这次分析中使用了什么方法、逻辑框架、判断依据？把"怎么做的"提炼出来
2. **DIKW 分流**：将过程中产生的信息按四层模型分别处理（见下表）
3. **不需要问用户"要不要存"**——方法论直接进大脑，这是你的职责，不是他的负担

### 具体实操：分析完成后自问三句

```
① 这次分析用了什么方法/框架？→ 存大脑（fact_store）
② 过程中产生了什么结构化数据？→ 存 entities/（write_file）
③ 有没有完整的分析报告？→ 存 vault/（write_file）
```

## 四、DIKW 四层模型——判断信息该放哪

```
┌─────────────────────────────────────┐
│         WISDOM（智慧）                │
│  不变的原则、框架、方法论              │
│  → 大脑（fact_store）                 │
├─────────────────────────────────────┤
│         KNOWLEDGE（知识）             │
│  实体信息、行业定义、概念解释            │
│  → 实体页（entities/）                │
├─────────────────────────────────────┤
│       INFORMATION（信息）              │
│  业务指标、快照数据——会变的动态数据      │
│  → Data Cache（带时间戳，30天失效）    │
├─────────────────────────────────────┤
│          DATA（数据）                  │
│  原始 API 返回、未处理的缓存            │
│  → Data/Cache 目录（临时）             │
└─────────────────────────────────────┘
```

**核心判断工具——问三个问题：**

| 问题 | 答案决定存哪 |
|------|-------------|
| 这个信息会变吗？ | **会变** → data cache（带时间戳） / **不会变** → 大脑或图书馆 |
| 我每次推理都要用吗？ | **高频** → 大脑（Holographic） / **偶尔查** → 图书馆（Vault） |
| 它是一张卡片还是一本书？ | **卡片**（单个实体）→ 实体页 / **体系化知识** → 图书馆 |

> ⚠️ 常见错误：把动态数据（快照、指标）塞进大脑。正确做法：动态数据进 data cache，大脑只存**不变的方法论**。
> ⚠️ 长文本存前先提炼成**一句话原则**再存（HRR 对短文本更友好）。

## 五、信息来了怎么办——三步分流

### Step 1：判断信息类型

```
信息进来
  │
  ├── 原始数据（未处理的API返回、日志）─────→ Data 临时目录
  │
  ├── 动态数据（指标/快照/短期判断）───────→ Data Cache（带时间戳）
  │
  ├── 结构化卡片（实体/概念/配置）────────→ 实体页 entities/
  │
  ├── 方法论/原则/红线（高频不变）────────→ 大脑 Holographic（高频用）
  │
  └── 完整报告/深度分析/部署文档 ─────────→ 图书馆 Vault（低频查）
```

### Step 2：检查有没有重复

- **有同类信息** → 更新/覆盖，不重复创建
- **没有** → 新建

### Step 3：执行存储

| 目标 | 用什么工具 | 示例 |
|------|-----------|------|
| 大脑 Holographic（存） | `fact_store(content="...", query="...", source="...")` | 存方法论、原则 |
| 大脑 Holographic（搜） | `fact_store(query="关键词")` — FTS5 + HRR 混合检索 | 搜历史记忆 |
| 大脑 Holographic（反馈） | `fact_feedback(action='helpful'/'unhelpful', fact_id=...)` | 校准信任评分 |
| 实体页 | `write_file` → `entities/{实体名}.md` | 写实体卡片 |
| 图书馆 | `write_file` → `vault/分类/文档名.md` | 写完整分析报告 |
| Data Cache | `write_file` → `data/{模块}/cache/` | 存临时数据 |

## 六、运行时检索——层级递进流水线

> 与"会话初始化"不同：初始化是冷启动补全背景，检索是每次用户提问时逐层查找。

用户提问时，按以下 **6 层递进检索，命中即返回，不继续下探**：

```
用户提问
  ↓
🏃 第 0 层：指代词快速路径
  "刚才/之前/上一条/你刚才说/刚刚" → session_search
  ↓
🧠 第 1 层：Holographic 双重检索
  自动：Hermes 内核每轮前自动 FTS5 检索注入（全局共享库，跨 session 命中）
  显式：fact_store(query="关键词1 关键词2") — HRR probe
  → 命中即返回
  ↓ 未命中（注意：中文长句可能因 FTS5 分词问题而假未命中，见本章末说明）
📦 第 2 层：缓存点检查
  检查 data/{模块}/cache/ 文件 mtime + TTL
  → 详见 cache-check skill
  ↓ 失效/无缓存
📚 第 3 层：知识库（Vault + 实体页）
  read_file entities/ → read_file vault/
  ↓ 未命中
🔍 第 4 层：近期对话
  session_search(关键词) — 换同义词再试
  ↓ 未命中
🌐 第 5 层：网络搜索
  web_search —— 最后手段
```

> **HRR 检索不命中时不要放弃**：换同义词再试 1-2 次。
> 不同关键词组合 = 不同的 HRR probe 结果，范围广的关键词更容易命中。

### 第 0 层：指代词快速路径 🏃
```python
if any(w in user_query for w in ["刚才", "之前", "上一条", "你刚才说", "刚刚"]):
    session_search(user_query)
    return
```

### 第 1 层：Holographic 检索 🧠
```
fact_store(query="关键词1 关键词2")
```
返回结果包含：`fact_id`, `content`, `trust_score`, `created_at`（跨 session，无隔离）。
→ 命中 → 回答
→ 未命中 → 换同义词再试，仍不中 → 进入第 2 层

> ⚠️ **Holographic 的检索局限**：HRR 代数是符号匹配，不是语义理解。
> 存了"市场高估时不建仓"，搜"牛市该不该买"可能不命中。
> **建议存事实时使用广谱关键词**（见「HRR 关键词编码策略」章节）。

> ⚠️ **⚠️ 中文检索假未命中**：如果你的 `fact_store(query="分批建仓策略")` 返回 0 条，但你知道数据库里有相关事实，
> 不要反复重试第 1 层——**直接跳到第 4 层 `session_search`**。
> 这是因为 FTS5 的 unicode61 tokenizer 把连续中文当作一个 token 造成检索遗漏，并非记忆库为空。
> **根本解决方案**见「FTS5 中文分词修复」章节（需要打一个补丁到 `store.py`）。

### 第 2 层：缓存点检查 📦

检查 `data/{模块}/cache/` 下是否有对应的缓存文件，根据**文件修改时间 + 数据类型有效期**判断是否可用：

| 数据类型 | 默认有效期 | 说明 |
|---------|-----------|------|
| 实时数据（API 响应） | 数小时内 | 根据数据更新频率调整 |
| 周频数据（评分/扫描） | 7 天 | 自然日 |
| 快照数据（配置/持仓） | 30 天 | 自然日 |
| 静态知识（方法论/规则） | 永久 | 不依赖缓存 |

**实操**：
```python
from pathlib import Path
import time

cache_file = Path("data/investment/cache/nav_data.json")
if cache_file.exists():
    age_hours = (time.time() - cache_file.stat().st_mtime) / 3600
    if age_hours < 24:  # 24小时内有效
        content = cache_file.read_text()
        # 直接用
    else:
        # 缓存过期，重新抓取
```

→ 缓存有效 → 直接用
→ 失效/无缓存 → 进入第 3 层

> ⚠️ **第 2 层是"数据快速通道"**：如果用户问的是刚抓过的最新数据（如今日净值），直接走缓存层，不要绕到 Holographic 转一圈。

### 第 3 层：知识库（Vault + 实体页）📚

搜索顺序：**先实体页，再文档库**。

**3a. 实体页（`vault/entities/`）**
```python
# 已知实体名 → 直接读
read_file("vault/entities/沪深300.md")

# 不确定路径 → 先搜文件名
search_files(pattern="*市盈率*", path="vault/entities/")
```

**3b. 文档库（`vault/分类/`）**
```python
# 按分类浏览
read_file("vault/工作/项目复盘.md")

# 全文搜索
search_files(pattern="复盘", path="vault/工作/")
```

**场景速查表**：
| 需求 | 优先去哪找 |
|------|-----------|
| 查一个指数/基金的定义 | `entities/` 找对应实体页 |
| 查一个系统的部署步骤 | `vault/系统运维/` 下的文档 |
| 查一个分析模型/框架 | `vault/方法/` 或 `vault/笔记/` |
| 不确定放哪了 | 先 `search_files` 搜内容 |

→ 命中 → 回答
→ 未命中 → 进入第 4 层

### 第 4 层：近期对话 🔍

通过 `session_search` 搜索历史会话记录：

```python
# 基本搜索
session_search("建仓策略 讨论")

# 精确短语（引号）
session_search('"左侧建仓 右侧确认"')

# 布尔逻辑（空格=AND, OR, NOT等）
session_search("高估 OR 泡沫 OR 不建仓")

# 搜索失败时：换同义词再试
session_search("买入 时机")   # 第一次搜
session_search("入场 时机")   # 不中，换同义词
session_search("等待 信号")   # 还不中，再换
```

**技巧**：
- 含指代词时（"刚才/上次"）先走第 0 层，不走这里
- 搜不到时换 2-3 组同义词组合再试
- `session_search` 只搜历史对话正文，不搜 Holographic 的 fact 库

→ 命中 → 回答
→ 未命中 → 进入第 5 层

### 第 5 层：网络搜索 🌐

最后手段，Holographic + 知识库 + 对话历史都找不到时才用。

```python
web_search("2025年 沪深300 PE 分位数")
web_search("恒生科技指数 当前PE 2025")
```

→ 找到 → 回答并告知来源
→ 找不到 → **诚实说"没找到，不在我的记忆库中"**

### 六条检索原则

1. **指代词优先**：含指代词先查近期对话，不绕圈子
2. **记忆优先**：之前聊过的事先查 Holographic，不瞎猜
3. **缓存优先**：刚抓过的数据先看 data cache，不重复抓
4. **实体页优先**：已知实体名先看 entities/，命中即秒回
5. **时效检查**：读 data cache 时要核实时效性
6. **诚实原则**：找不到就说找不到，不编造。Holographic 检索可能漏召回，可以再试 `session_search`

## 七、FTS5 中文分词修复（⭐ 解决中文检索问题的完整方案）

> ⚠️ **这是本指南最重要的章节。** FTS5 默认的 unicode61 tokenizer 在中文场景下有一个严重缺陷：
> 它把连续的中文字符（CJK 统一表意文字区间）当做一个 token 处理，导致中文子串搜索完全失效。
> 本文给出一个**零外部依赖、零架构变更**的修复方案，基于 Hermes 0.14 插件体系实测。

### 7.1 问题现场

| 场景 | 存进去的内容 | 检索词 | 预期结果 | 实际结果 |
|------|------------|--------|---------|---------|
| 存了一个投资原则 | "分批建仓需要等待动量信号确认" | `"建仓"` | ✅ 命中 | ❌ 0 条（FTS5 把整句当 1 个 token） |
| 存了基金名 | "南方标普红利低波50ETF联接A" | `"红利"` | ✅ 命中 | ❌ 0 条 |
| 搜索策略 | "买入时机 信号 加仓" | `"加仓"` | ✅ 命中 | ❌ 0 条（即使你刻意加了空格，unicode61 也不分割全中文） |
| 纯英文 | "Never buy during a bubble" | `"bubble"` | ✅ 命中 | ✅ 5 条（英文按空格分词正常） |

**根因**：FTS5 的 `unicode61` tokenizer 对 CJK 字符的处理方式是**将连续的中文字符视作一个 token**。这是其设计特性，不是 bug——它不知道中文词边界在哪里，所以安全起见不分词。

**数学上**：
```
"分批建仓策略" → unicode61 分词后 → ["分批建仓策略"]（1个token）
"分批"、"建仓"、"策略" 都不单独作为 token 存在 → 搜索这些子串均无法命中
```

### 7.2 解决方案：字符间距法（零外部依赖）

**核心思路**：在 CJK 字符（U+4E00-U+9FFF）之间插入空格，使 unicode61 能够逐字分词。

```
修复前："分批建仓需要等待动量信号确认"
修复后："分 批 建 仓 需 要 等 待 动 量 信 号 确 认"
```

搜索时也一样用空格分隔中文字符：`fact_store(query="建 仓")` 即可命中。

**这行得通的数学依据**：
- unicode61 的默认分词规则是**以空格/标点/ASCII 边界分割**
- CJK 字符间插入空格后，unicode61 把它们当作独立 token
- FTS5 的短语匹配 `"建 仓"` 要求相邻 token 依次出现 → 相当于精确子串匹配
- 英文不受影响（英文单词本来就以空格分隔）

### 7.3 实现代码

在你的 `memory_store.db` 所在环境下的 Holographic 插件源码中，添加以下函数并修改 FTS5 读写路径。

**函数定义**（添加到 `store.py` 中）：

```python
import re

# CJK 统一表意文字区间
_CJK_PATTERN = re.compile(r'[\u4e00-\u9fff]')

def _space_cjk(text: str) -> str:
    """在 CJK 字符间插入空格，使 FTS5 unicode61 能逐字分词。
    
    非中文文本原样返回（零开销）。 三步处理：
    1. 连续 CJK 之间插入空格："分批" → "分 批"
    2. CJK 与 ASCII 之间也插入空格："PE分位" → "PE 分 位"
    3. ASCII 与 CJK 同样："买入ETF" → "买 入 ETF"
    """
    if not _CJK_PATTERN.search(text):
        return text
    result = re.sub(r'([\u4e00-\u9fff])(?=[\u4e00-\u9fff])', r'\1 ', text)
    result = re.sub(r'([\u4e00-\u9fff])([^\u4e00-\u9fff\s])', r'\1 \2', result)
    result = re.sub(r'([^\u4e00-\u9fff\s])([\u4e00-\u9fff])', r'\1 \2', result)
    return result
```

**需要改动的关键位置**，共 6 处（以下行号对应 Hermes 0.14 官方版，你的版本可能略有不同）：

| 位置 | 改动 | 作用 |
|------|------|------|
| `store.py` 新增函数 | 添加 `_space_cjk()` | 核心函数 |
| `add_fact()` 中写 content 前 | `fts_content = _space_cjk(content)` | 写入时空格化 |
| `search_facts()` 中 query 解析 | `spaced = _space_cjk(query)` | 搜索时空格化 |
| `update_fact()` 中写 content | `fts_content = _space_cjk(content.strip())` | 更新时空格化 |
| `remove_fact()`（可选） | 无需改动 | 删除不需要空格化 |
| FTS5 重建方法 | 用 `_space_cjk()` 遍历全库重填 FTS5 表 | 迁移已有数据 |

### 7.4 已有数据迁移

如果 DB 中已有大量事实（几千条），需要重建 FTS5 索引。**不要删除原 facts 表**——只重建 FTS5 虚拟表。

**一步到位方案**（在 Python 中执行）：

```python
import sqlite3
import re

_CJK_PATTERN = re.compile(r'[\u4e00-\u9fff]')

def _space_cjk(text: str) -> str:
    if not _CJK_PATTERN.search(text):
        return text
    result = re.sub(r'([\u4e00-\u9fff])(?=[\u4e00-\u9fff])', r'\1 ', text)
    result = re.sub(r'([\u4e00-\u9fff])([^\u4e00-\u9fff\s])', r'\1 \2', result)
    result = re.sub(r'([^\u4e00-\u9fff\s])([\u4e00-\u9fff])', r'\1 \2', result)
    return result

conn = sqlite3.connect('/home/zhao/.hermes/memory_store.db')
conn.execute("PRAGMA journal_mode = WAL")
conn.execute("PRAGMA synchronous = OFF")

# 删除旧 FTS5 表并重建
conn.execute("DROP TABLE IF EXISTS facts_fts")
conn.execute("CREATE VIRTUAL TABLE facts_fts USING fts5(content, tags)")
conn.commit()

# 遍历所有 facts，空格化后写入 FTS5
rows = conn.execute("SELECT id, content, tags FROM facts").fetchall()
print(f"Migrating {len(rows)} facts...")
for row_id, content, tags in rows:
    fts_content = _space_cjk(content if content else "")
    conn.execute(
        "INSERT INTO facts_fts (rowid, content, tags) VALUES (?, ?, ?)",
        (row_id, fts_content, tags)
    )
conn.commit()
print("Done. FTS5 rows:", conn.execute("SELECT COUNT(*) FROM facts_fts").fetchone()[0])
conn.close()
```

**⚠️ 迁移前做 3 件事**：
1. **备份** `cp ~/.hermes/memory_store.db ~/.hermes/memory_store.db.bak`
2. **停服务**：`systemctl --user stop hermes-gateway` 或 `kill <gateway-pid>`，防止并发写入
3. **确认 numpy 已安装**（HRR 需要）：`python3 -c "import numpy; print('OK')"`

### 7.8 进阶加固：三层 CJK 检索回退

**仅做 `_space_cjk()` 修复后**，中文搜索已经可用（写和查都加空格即可）。但用户如果直接输入未空格的中文 query（如 `fact_store(query="分批建仓")`），仍可能搜不到。

**本地生产环境已验证的加固方案**——在 `search_facts()` 中加入三层回退机制：

#### 第 1 层：FTS5 短语匹配（精确）
```python
# CJK query → 用空格化后的短语精确匹配
spaced = _space_cjk(query)
fts_query = f'"{spaced}"'    # 如 "建 仓 策 略"
```
命中条件：FTS5 token 按空格化后的顺序**连续出现**。

#### 第 2 层：FTS5 AND 匹配（宽松）
```python
# 如果短语匹配返回 0 条 → 用 AND 松散匹配
and_terms = " AND ".join(spaced.split())  # "建 AND 仓 AND 策 AND 略"
```
命中条件：所有 CJK 逐字 token 都出现在同一行，不要求顺序。

#### 第 3 层：字符 Bigram Jaccard 相似度（模糊）
```python
# 字符 2-gram 重叠度匹配（预计算索引）
# fact_bigrams 表存每个事实的 CJK 字符 2-gram
# 查询时取 query 的 2-gram，与索引做 Jaccard 分数
# 阈值 ≥0.2（20% 重叠）即召回
```

**需要新增的数据库表和索引**（在 `__init__` 的建表语句中添加）：
```sql
CREATE TABLE IF NOT EXISTS fact_bigrams (
    bigram  TEXT    NOT NULL,
    fact_id INTEGER NOT NULL REFERENCES facts(fact_id),
    PRIMARY KEY (bigram, fact_id)
);
CREATE INDEX IF NOT EXISTS idx_bigram_lookup ON fact_bigrams(bigram);
```

**需要新增的方法**（添加到 `MemoryStore` 类）：

- **对于零基础的用户**：只做 `_space_cjk()` + FTS5 重建即可满足 90% 需求（写和查都记得加空格）。
- **对于进阶部署**：建议加上 bigram 表和三层回退（增加约 20 行代码 + 一个 SQL 表），可以**完全不用关心写/查时是否加空格**。

### 7.9 已知限制与边界情况

| 边界情况 | 行为 | 建议 |
|---------|------|------|
| 纯英文文本 | 完全不受影响 | 照常使用 |
| CJK + 英文混合（"PE分位"） | 自动拆分 → `"PE 分 位"` | 正常 |
| 标点符号（逗号、引号等） | 不受影响，unicode61 自动忽略 | 正常 |
| 纯标点 CJK（"！？"） | 不受影响 | 正常 |
| FTS5 短语查询 `"精 确 匹 配"` | **必须精确顺序匹配** | 用单字搜索更灵活 |
| FTS5 前缀查询 `"建"*` | 可用，但不太需要（已逐字分词） | 通常用 `"建 仓"` 即可 |
| 用户通过 `fact_store(query=...)` 搜索 | **query 参数也要空格化** | 见 7.7 节 |
| 已有旧数据 | 空间化后 FTS5 索引重建即可 | 见 7.4 节 |
| 没有 numpy 的环境 | 不影响（FTS5 是独立组件） | 但 HRR 会降级，见配置章节 |

### 7.9 为什么不采用其他方案

| 方案 | 评估结果 | 原因 |
|------|---------|------|
| FTS5 trigram tokenizer | ❌ | 需重新编译 SQLite 或加载扩展，嵌入式环境不可行 |
| FTS5 ICU tokenizer | ❌ | 需要 ICU 扩展库（~50MB），ARM64 编译困难 |
| jieba 分词 | ❌ | 增加依赖，词库加载慢，过度设计 |
| 转 `LIKE` 查询替代 FTS5 | ❌ | 全表扫描慢 ~19 倍，放弃索引优势 |
| 引入语义搜索（bge-m3） | ❌ | 增加 ~2GB 内存 + CPU 负载 + 网络依赖，背离轻量原则 |
| **字符间距法 ✅** | ✅ | 零依赖、零架构变更、0.1ms 查询、100% 命中率 |

## 八、Holographic 配置调优

> Holographic 默认权重（FTS5=0.4, Jaccard=0.3, HRR=0.3）适合通用场景。
> 如果搜索效果不理想，可以在配置层面调整。

### 配置（config.yaml）

```yaml
plugins:
  hermes-memory-store:                          # ← 注意：key 名是 hermes-memory-store
    db_path: ~/.hermes/memory_store.db
    auto_extract: 'false'                       # 依赖 SOUL.md 三问流程，不自提取
    default_trust: '0.5'                        # 新事实信任分（中性推荐）
    min_trust_threshold: '0.3'                  # 检索过滤下限
    temporal_decay_half_life: '0'               # 关闭时间衰减（方法论不过期）
    hrr_weight: 0.15                            # HRR 检索权重（中文场景推荐 0.15）
    # hrr_dim: 1024  ← 默认值，无需显式设置
```

> ⚠️ **配置 key 注意**：Holographic 的配置键是 `plugins.hermes-memory-store`，不是 `plugins.holographic`。

### 权重含义

| 权重 | 默认值 | 作用 | 配置键 |
|------|--------|------|--------|
| FTS5 | 0.4 | 关键词全文检索（精确命中） | ❌ 不可配置（硬编码） |
| Jaccard | 0.3 | 词元重叠度重排（容错模糊匹配） | ❌ 不可配置（硬编码） |
| HRR | 0.3 | 代数相关性检索（灵活近义匹配） | ✅ `hrr_weight` |

> ⚠️ Holographic 没有 `textWeight` / `vectorWeight` / `hybrid_weight` / `retrieval_strategy` / `minScore` 这些配置项——那是 Hindsight 或其他 provider 的。实际可调项只有 `hrr_weight`。

### 调优指引

**场景 1：中文文档/精确查找**
→ 降低 `hrr_weight`（减小代数匹配的噪声）
```yaml
hrr_weight: 0.15   # FTS5 + Jaccard 占主导
```

**场景 2：灵活近义搜索（口语化/模糊描述）**
→ 保持或提高 `hrr_weight`
```yaml
hrr_weight: 0.4    # HRR 代数匹配占比更高
```

**场景 3：无 numpy 时自动降级**
如果 numpy 不可用，HRR 权重**自动归零**，并重新分配给 FTS5=0.6, Jaccard=0.4：
```
无 numpy → hrr=0.0, fts=0.6, jaccard=0.4
```
此时配置中的 `hrr_weight` 设置会被忽略，检索完全依赖 FTS5 + Jaccard。

### DeepSeek 建议中不适用于 Holographic 的项

| 建议 | 为什么不适用 |
|------|-------------|
| 固定 session ID | Holographic 全局共享，无需 session 隔离 |
| `retrieval_strategy: hybrid` | 无此配置项 |
| `minScore` 阈值调低 | 无此配置项 |
| `hermes memory rebuild --force` | Holographic 无向量索引可重建 |
| 迁移到向量数据库 (ChromaDB/FAISS) | 轻量设备资源有限，HRR + FTS5 够用 |
| 集成 GHRR | 纯学术概念，无 Holographic 实现 |

> **总结**：Holographic 的调优核心只有**两个有效杠杆**：
> 1. `hrr_weight` — 权重分配
> 2. **HRR 关键词编码质量**（见下一章）——这才是真正起决定性作用的

## 九、HRR 关键词编码策略

> ⚠️ 以下是最关键的优化点——**HRR 检索的成败 90% 取决于 query 参数的编码质量**。

### 核心公式

```
query = 核心概念 + 同义词群 + 近义词群 + 相关场景词
```

### 好例子 vs 差例子

| | query 参数 | 效果 |
|--|-----------|------|
| ❌ | `"建仓"` | 用户只说"进场"就搜不到 |
| ✅ | `"建仓 买入 入场 加仓 进场 开仓 买进"` | 无论哪种说法都能命中 |
| ❌ | `"高估"` | 搜"估值过高 泡沫 贵了"都不中 |
| ✅ | `"高估 估值过高 泡沫 太贵 不便宜 高估值"` | 全面覆盖 |

### 分类编码模板

**投资原则类：**
```python
query = " ".join([
    # 核心概念
    "建仓 加仓 买入 入场",
    # 反面词（HRR 能识别反面关系）
    "不减仓 不卖出 持有",
    # 条件/场景
    "时机 等待 确认 信号 右侧 左侧",
    # 情绪/判断
    "保守 谨慎 观望 耐心",
])
```

**用户偏好类：**
```python
query = " ".join([
    "报告 推送 通知",
    "偏好 喜欢 习惯 方式",
    "接收 发送 格式 排版",
])
```

### 关键词编码要诀

1. **广度优先**：宁多勿少，10 个同义词比 3 个精确词好
2. **场景覆盖**：把可能问到的场景词都放进去
3. **使用自然语言中的常用词**：用户大概率怎么说，就怎么编
4. **存事实时顺便编 query**：每次 `fact_store` 时，花 3 秒想一想"以后我会怎么搜这个"
5. **定期补 query**：发现某个事实总搜不到，用 `fact_feedback` + 重新 `fact_store` 更新 query

**检索时的技巧：**
```
搜不到的时候，换以下思路各试一次：
1. 缩小范围：加更多限定词
   fact_store(query="建仓 买入 等待")
2. 扩大范围：去限定词，留核心词
   fact_store(query="建仓")
3. 完全换一套词：换个角度描述同一概念
   fact_store(query="入场 时机 右侧")
```

## 十、时效性管理——什么信息会过期

| 信息类型 | 有效期 | 过期后怎么办 |
|---------|--------|------------|

## 十四、信任分自动校准（反馈闭环）

> 本系统是「三层自愈闭环」的第三环：`agent/fact_feedback_loop.py` + cron。

### 14.1 为什么需要自动校准

`fact_feedback` 工具可以让 Agent 手动标记事实是否准确（`action='helpful'` / `action='unhelpful'`），但有两个致命问题：
1. **Agent 经常忘记手动标记** — 检索后没调 `fact_feedback`
2. **大量事实长期无人问津** — 陈旧事实占用检索槽位

### 14.2 核心规则（直接操作 SQLite）

| 规则 | 条件 | 动作 | 含义 |
|------|------|------|------|
| 降权 | retrieval_count ≥ 3 AND helpful_count = 0 | trust -= 0.1 | 被检索 3+ 次但从无帮助 → 低质量 |
| 升权 | helpful / retrieval > 30% | trust += 0.05 | 检索后被标记有帮助的比率高 → 高质量 |
| 衰减 | created_at > 14天 AND retrieval = 0 | trust *= 0.95 | 超过两周无人问津 → 低价值 |
| 保护 | — | trust ∈ [0.05, 0.95] | 防止极端值 |

### 14.3 使用方式

```bash
# 查看健康报告
python3 -m agent.fact_feedback_loop --report

# 试运行（不实际写入）
python3 -m agent.fact_feedback_loop --calibrate

# 实际校准
python3 -m agent.fact_feedback_loop --calibrate --apply
```

### 14.4 自动调度

cron `fact-feedback-calibrate` 每周六 11:00 自动执行：dry-run → 评估 → apply → 推送报告。

### 14.5 与手动 `fact_feedback` 的关系

| 维度 | 手动 `fact_feedback` | 自动 `fact_feedback_loop.py` |
|------|---------------------|---------------------------|
| 触发方式 | Agent 自觉 | cron 定时 |
| 判断依据 | 语义准确性（"这条真的有用吗"） | 统计规则（调用次数 + 帮助比 + 时效） |
| 适用范围 | 单条具体事实 | 全库批量处理 |
| 频次 | 每次检索后 | 每周一次 |
| 谁是主导 | Agent 判断上界 | 系统维护下界 |

```
手动标记 → 累加 helpful_count/retrieval_count → 自动校准规则触发 → trust 调整
```

### 14.6 首次部署

```bash
cp agent/fact_feedback_loop.py ~/.hermes/hermes-agent/agent/
python3 -m agent.fact_feedback_loop --calibrate --apply
```

## 十五、MEMORY.md 容量管理（DIKW 自动分流）

> 本系统是「三层自愈闭环」的第二环：`memory-capacity-management` skill + cron。

### 15.1 背景

`MEMORY.md` 有字符上限（通用版 ~6000 字）。超过 80% 时：
- 新增无法写入
- system prompt 中记忆占比过大挤占推理空间
- 旧的、低频的信息与高频原则抢位置

### 15.2 自动分流流程

```
MEMORY.md 占用 >85% → cron 触发（每周五 10:00）
  ├─ 逐条读 MEMORY.md
  ├─ DIKW 三问：
  │    会变吗？→ data cache
  │    高频用？→ fact_store（大脑）
  │    卡片 vs 书？→ entities / vault
  └─ 存储后替换为一行索引
     [分类-关键词] | 一句话 + 详见路径
```

### 15.3 分流规则详解

| 信息类型 | 存哪 | 目标位置 | 格式 |
|---------|------|---------|------|
| 用户偏好/原则 | 🧠 大脑 | `fact_store(content=..., query=..., source="user_pref")` | 短句 ≤50 字 |
| 方法论/工作流 | 🧠 大脑 | `fact_store(content=..., query=..., source="general")` | 短句 + 关键词群 |
| 项目详情/配置 | 📚 图书馆 | `vault/` | 完整文档 |
| 实体信息 | 🗂️ 卡片柜 | `vault/entities/` | 结构化卡片 |
| 临时数据 | 📦 仓库 | `data/{模块}/cache/` | 带时间戳的原始数据 |
| 大段过期信息 | 🗑️ 删除 | — | `memory(action='remove')` |

### 15.4 手动触发

用户消息包含以下 triggers 关键词之一时，`skill_auto_trigger` 自动注入 skill：
- 整理记忆、memory满、容量管理、收拾记忆、清理记忆、MEMORY满了、记忆满了、记忆快满了、记忆整理

### 15.5 索引行格式示例

```markdown
- [投资-调仓决策20260602] | 方案A：减013309 10%→17.5%，加001632 2.5%→5% | 详见 vault/...
- [系统-temperature机制] | MiniMax temperature 已从代码删除 | 详见 vault/...
```

## 十六、技能自动触发（运行时层）

> 本系统是「三层自愈闭环」的第一环：`agent/skill_auto_trigger.py` + 9 行补丁。

### 16.1 是什么

在 `conversation_loop.py` 的用户消息构造前注入钩子，自动扫描 `skills/**/SKILL.md` 的 frontmatter 中 `triggers` 字段，当用户消息包含匹配关键词时，将 skill 完整内容自动注入到 LLM 上下文中。

### 16.2 工作流程

```
用户说"整理记忆" → conversation_loop 收到消息
  → skill_auto_trigger.auto_load_triggered_skills("整理记忆")
     ├─ 扫描 ~/.hermes/skills/**/SKILL.md 的 triggers 字段
     ├─ 匹配 memory-capacity-management（triggers: 整理记忆）
     └─ 返回该 SKILL.md 内容
  → 拼接成 skill_content + "\n\n---\n\n" + user_message
  → LLM 拿到 skill 指令 + 原问题
```

### 16.3 SKILL.md 格式要求

```yaml
---
name: my-awesome-skill
triggers: [关键词1, 关键词2, 整理记忆]
---
```

有 `triggers` 的 skill 自动生效，无 `triggers` 的不受影响。

### 16.4 核心设计原则

- **Best-effort**：整个钩子包裹在 `try/except` 中，失败也不影响对话
- **纯关键词匹配**：`keyword in user_message.lower()`，简单可靠
- **缓存提升性能**：首次扫描后缓存技能列表和文件内容，后续 O(1) 匹配
- **缓存失效**：`clear_trigger_cache()` 在技能文件发生变化时重置

## 十八、源码级 DIKW 强制——vault-write 自动提炼

> 本系统是「四层自愈闭环」的第四环：`agent/tool_executor.py._inject_vault_duty()` 源码钩子。

### 18.1 背景

前三环（自动触发 → 容量管理 → 反馈校准）覆盖了「用户说出关键词时注入 skill」「MEMORY.md 满了自动分流」「信任分自动衰减」。但存在一个系统级盲区：

**Agent 写了完整文档到 vault，但忘记提炼方法论到 fact_store。**

之前的解法是在 SOUL.md（system prompt）和 SKILL.md 中各写了一条"写完必提"规则。问题是：规则可以被遵守，也可以被遗忘——Context 窗口一长，Agent 可能跳过。

### 18.2 源码级解法

在 `agent/tool_executor.py` 中新增 `_inject_vault_duty()` 函数，hook 到**每一次工具执行的 tool result 追加之后**：

```python
def _inject_vault_duty(function_name: str, function_args: dict, messages: list) -> None:
    """If the tool just wrote to a vault path, inject the extraction duty."""
    if function_name != "write_file":
        return
    _path = function_args.get("path", "") or ""
    if not _path:
        return
    _normalized = _path.replace(os.sep, "/")
    if "vault" not in _normalized.split("/"):
        return
    current = messages[-1].get("content", "")
    if isinstance(current, str):
        messages[-1]["content"] = current + _VAULT_DUTY_NOTE
```

注入的 duty 内容：
```
[DUTY: 你刚刚将完整文档写入 vault。根据 DIKW 模型，你必须在此后自动提炼 1-3 条核心方法论，
用事实库（fact_store）存入大脑。这是源码级强制规则，不依赖用户提醒。]
```

### 18.3 为什么是源码级

| 方案 | 触发时机 | 可靠性 |
|------|---------|--------|
| SOUL.md 规则（system prompt） | 每次对话初始化时加载 | ~70%（可能被上下文压缩删掉或 Agent 忽略） |
| SKILL.md 规则（dikw-memory-flow） | 用户消息含"记住"等关键词时自动触发 | ~80%（触发后 Agent 可能不读完整） |
| **源码级 _inject_vault_duty()** | **write_file 到 vault/ 后直接在 tool result 尾部注入** | **100%**（LLM 下一次回复必然看到 tool result） |

### 18.4 触发路径

```
Agent 调用 write_file(path="vault/xxx")
  → tool_executor 执行工具
  → make_tool_result_message(...) 追加到 messages
  → _inject_vault_duty() 检查路径 → 命中 vault/
  → messages[-1]["content"] += "[DUTY: 必须提炼方法论到 fact_store]"
  → LLM 收到 tool result（含 DUTY 提醒）
  → LLM 生成回复时自动执行 fact_store 提炼
```

### 18.5 与手动 DIKW 的关系

- **这是兜底层，不是替代层**：Agent 仍然应该在分析完成后自觉执行 DIKW 三问分流
- **只覆盖 `write_file`**：`patch`、`terminal` 中的手动写入不会触发
- **只检查路径含 `vault/` 组件**：写在其他地方（cache、/tmp、data）不触发
- **只提炼到 fact_store（大脑）**：不自动写入 vault/entities/ 或 MEMORY.md（需要 Agent 自己判断完整分流）

### 18.6 部署方式

```bash
# 定位到 tool_executor.py 中 messages.append(make_tool_result_message(...)) 行
# 在这两行之后各插入 _inject_vault_duty(fn_name, fn_args, messages)：

# 顺序路径（~L893）
# → 在 make_tool_result_message 下一行加
_inject_vault_duty(function_name, function_args, messages)

# 并发路径（~L474）
# → 在 make_tool_result_message 下一行加
_inject_vault_duty(name, args, messages)
```
